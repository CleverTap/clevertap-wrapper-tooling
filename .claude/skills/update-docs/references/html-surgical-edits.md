# Editing the generated HTML surgically

How to change a `docs/_generated/*.html` page **in place** so it matches a Markdown edit, without
re-flowing or regenerating the page. Read this when doing step 4 (and 5) of the skill.

## The golden rule

> **Change only the lines the edit requires. Everything else stays byte-identical.**
> After editing, `git diff docs/_generated/<page>.html` (it's gitignored, so use
> `git diff --no-index <backup> <page>` or just review visually) must show **only** the intended
> lines. If your diff is large, you regenerated by mistake — undo and edit narrowly.

## 1. Find the right page, then the right spot

- **Page:** map the Markdown page to its HTML file via the file-map table in
  `docs/_generated/_site-spec.md` (e.g. `docs/20-walkthroughs/diff-native-api/02-source-acquisition.md`
  → `docs/_generated/walk-diff-02-source-acquisition.html`).
- **Spot:** locate the section by its **heading text** (the HTML keeps the same headings as the MD),
  or by a unique string from the code/table you're changing. Edit that block only.

## 2. The HTML shapes (match these exactly when editing)

These are the components the pages use. When you change content, keep the surrounding markup.

**Code block** (the annotated-code cards):
```html
<div class="code-card">
  <div class="code-card__head"><span class="code-card__lang">python</span> …optional filename…</div>
  <pre><code> …code, with the ①②③ markers as plain text… </code></pre>
</div>
```
- Edit the text inside `<pre><code>…</code></pre>`. Keep HTML-escaping: `<`→`&lt;`, `>`→`&gt;`,
  `&`→`&amp;`. Preserve the circled markers (①②③…). Don't add traffic-light dots.

**Annotation table** (marker → meaning):
```html
<div class="table-wrap"><table> … <tr><td>①</td><td>…</td><td>…</td></tr> … </table></div>
```
- Edit/add/remove `<tr>` rows to match the code's markers.

**Callouts** (map from the Markdown markers):
```html
<div class="callout callout--info">  <span class="callout__label">…</span> … </div>
```
- 🟦 beginner sidebar → `callout--info`; 🧠 analogy → `callout--analogy` (or `--amber`);
  ⚠️ warning → `callout--warn`; plain note → `callout--note`. Use the variant already used on the page.

**Check-yourself / Q&A folds:**
```html
<details class="qa"><summary>Question?</summary><div class="qa-body"> Answer… </div></details>
```
- Keep this structure so it stays foldable. Add/edit a `<details class="qa">` per question.

**Page navigation** (top + footer) — usually leave untouched; only edit if you renamed/reordered pages:
```html
<nav class="kit-nav"> <a href="index.html">◀ Onboarding Kit</a> <span class="crumb">…</span> <a href="NEXT.html">Next ▶</a> </nav>
…
<footer class="kit-foot"> <a href="PREV.html">◀ …</a> <a href="index.html">⌂ Kit home</a> <a href="NEXT.html">… ▶</a> </footer>
```
If you add/remove/reorder pages, update the prev/next here **and** the file-map in `_site-spec.md`
**and** the TOC in `index.html`.

## 3. Diagrams — render, then inject (never hand-write SVG)

A diagram lives in this shell:
```html
<div class="diagram-shell">
  <div class="mermaid-wrap">
    <div class="zoom-controls"> …buttons… <span class="zoom-label">…</span></div>
    <div class="mermaid-viewport">
      <div class="mermaid mermaid-canvas"> …THE INLINE &lt;svg&gt; LIVES HERE… </div>
    </div>
  </div>
  <script type="text/plain" class="diagram-source"> …the Mermaid source text… </script>
</div>
```
When a diagram's source changed:
1. Update the Mermaid text in **two** places: the `.mmd` / ```mermaid block in the Markdown, **and**
   the `<script type="text/plain" class="diagram-source">` in the HTML (kept for reference/regeneration).
2. Render the new source to SVG via `mcp__claude_ai_Mermaid_Chart__validate_and_render_mermaid_diagram`
   (confirm `valid:true`; HTML-escape literal `<...>` only in the copy you send the renderer if it
   throws `TAGSTART`). Save the `rawSVG` to a file via a script — **don't read the SVG into context.**
3. Replace the **entire inline `<svg>…</svg>`** inside `.mermaid-canvas` with the new SVG (do this with
   a script reading from the saved file, so the big string never enters your context).

**Never:**
- hand-author or hand-tweak SVG markup;
- reintroduce a CDN loader — there must be **no** `cdn.jsdelivr`, no `import mermaid`, no
  `<script type="module">` (that's the bug that made diagrams hang on "Loading…"). The page renders
  diagrams purely from the inlined SVG; the loader is the offline `render()` that reads the existing
  `<svg>` (see `docs/90-meta/how-this-kit-is-built.md`).

## 4. Don't confuse Mermaid's internal classes with page classes

Inside an inlined `<svg>` you'll see Mermaid's own classes: `node`, `marker`, `label-container`,
`flowchart`, `basic`, `root`, `nodes`, `edgePaths`, etc. **These are part of the rendered diagram —
don't edit them by hand.** To change a diagram, change its source and re-render (section 3). The
page-level components you *do* edit are the ones in section 2 (`code-card`, `callout`, `qa`,
`table-wrap`, `kit-*`).

## 5. After editing
Run the step-6 verification from `SKILL.md`: dead-link check, `grep` for `cdn.jsdelivr`=0 and
`type="module"`=0, confirm the touched diagram page still has an inline `<svg`, and that any changed
Mermaid validates. Then log to `docs/90-meta/fact-check-log.md`.
