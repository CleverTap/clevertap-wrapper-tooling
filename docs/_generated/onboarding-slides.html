Onboarding · from zero

wrapper‑sync

The one-button automation that keeps CleverTap wrappers in step with the native SDK

A native SDK ships → a maintainer clicks one button → a robot reads what changed, edits the wrapper, and opens a pull request for a human to review.

The problem

Every native release leaves the wrappers behind.

CleverTap's native SDKs — Android (Kotlin/Java) and iOS (Obj-C/Swift) — release on their own schedule: bug fixes, new features, new public methods.

But many apps are cross-platform (Cordova, Flutter, React Native). They reach the native SDK through a thin wrapper SDK. When the native SDK moves, every wrapper has to catch up.

Catch-up checklist

  • Bump the version pin the wrapper points at
  • Expose new methods through the bridge — JS + Android + iOS
  • Carry over build changes (a permission, a higher min Android)
  • Update the changelog and the example apps
  • Build everything to confirm it still compiles

Why automate it

Done by hand, for three wrappers, every release.

  • Tedious and easy to get wrong. A method gets missed. A version gets bumped in four places but not the fifth.
  • Exactly the careful, repetitive work that humans are bad at and robots are good at.
Analogy

The native SDK is a dictionary that just added new words. Each wrapper is a translation of it. When the dictionary grows, every translation must be updated — or it falls out of date.

The one-button promise

Click once, pick the versions, walk away — minutes later a pull request appears.

The robot reads what changed, makes the edits, builds the app to check its own work, and writes a tidy summary. The output is always a PR — never a silent change.

It's a dishwasher, not a maid: you still load it and put the dishes away (review + merge) — you're just not scrubbing each plate by hand.

01

The one model that explains everything

The 4-Layer Onion

A click travels down the layers — Trigger → Conductor → Brain → Ground Truth. A finished pull request travels back up.

Layer ① · Trigger

A button in the wrapper repo.

  • Where: a tiny native-release-sync.yml in each wrapper repo.
  • What it is: a form. Pick the native versions (e.g. Android core → 8.2.0), click Run workflow.
  • What it does: almost nothing itself — it just calls the shared workflow in the hub and passes versions + four secrets.
Analogy

A vending-machine button. Pressing it doesn't make the snack — it tells the machine behind the glass to do that.

Layer ② · Conductor

sync.yml in the hub repo.

  • Setup — mint a bot token, check out the wrapper, make a fresh branch.
  • Pre-sync build gate — build the example app first; if it's already broken, stop (no AI spend).
  • Sync Android, then iOS — hand the real work to Layer ③.
  • Post-sync build — rebuild with the edits to see if they compile.
  • Commit, push, open the PR — then report cost / notify on failure.
Analogy

A recipe checklist: "preheat; if the batter rose, frost it; if it burned, throw it out." Its cleverness is gating — not domain knowledge.

The single most important idea

“
The YAML does not wrap any APIs. It just hands a written prompt to an AI — and the AI does the work.
Layer ② is plumbing · Layer ③ is intelligence

Layer ③ · Brain

Claude, running headless, does the actual wrapping.

  • Reads the cold facts from Layer ④, then a recall pass over the changelog.
  • Triages each change: surface / skip / defer / flag.
  • Writes the bridge code on all sides — JS, Android, iOS.
  • Bumps every version, updates the changelog, adds example-app demos.
  • Emits a structured summary — plus the run cost.
Analogy

Claude is the chef. The conductor handed it the order ticket (the prompt); the chef decides how to cook and cooks — but only with real ingredients.

Layer ④ · Ground Truth

diff_native_api.py

The one deterministic, non-AI component. Given two native versions, it reports exactly what changed: methods added/removed/changed, build-setting moves, the relevant changelog text. No opinions.

An AI can hallucinate; this tool cannot. So Claude is told to trust it as fact and to verify any method in real source before using it.

Honest limit: regex finds ~80% of changes; the recall pass over the changelog catches the rest.
Librarian vs editor

The diff tool is the librarian who only tells you what's actually on the shelf. Claude is the editor who decides what to publish.

You want the editor creative — but anchored to a librarian who never makes things up.

End to end

One click → one pull request.

The onion in motion

A Day in the Life

One real sync, narrated. Android core just released 8.2.0; the wrapper is pinned at 8.1.0. iOS isn't in this release — so the maintainer syncs Android only.

The run · part 1 of 3

Click → setup → the gate.

SCENE 1
The click
Maintainer fills the form: android_module = core, android_version = 8.2.0, ios_module = none. Clicks the green button. That's the whole human effort until the PR.
Layer ① Trigger
SCENE 2
Setup
Conductor mints a short-lived bot token (a GitHub App acts without a personal account), checks out the wrapper, creates a fresh task/release_* branch, installs the Claude CLI.
Layer ② Conductor
SCENE 3
Pre-sync gate
Build the example app as-is, before any change. If it's already broken, the run stops here and $0 of AI is spent. In our run it's green — continue.
fail fast, fail cheap

The run · part 2 of 3

Scene 4 — Claude syncs Android.

  • Runs the diff tool for core 8.1.0 → 8.2.0 — cold facts (Layer ④).
  • Reads it as ground truth, then a recall pass over the changelog (~20% regex misses).
  • Triages every change: surface / skip / defer / flag.
  • Source-verifies before writing — "verify or flag, never guess."
  • Writes the bridge (JS/Android/iOS), bumps versions, updates changelog + examples, emits a summary.
Why "never guess"

Guessing a method that doesn't exist = a broken build. So Claude must find the real method/selector in source first — or flag it for a human.

The run · part 3 of 3

Post-sync build → PR → merge.

SCENE 5
Post-sync build
Rebuild on Claude's edits. A failure is non-fatal — the PR still opens with a build-failed label and a banner for a human to fix. Ours is green.
failure ≠ stop
SCENE 6
Open the PR
Commit + push; Claude writes the PR body (surfaced / skipped / deferred / flagged + changelog + cost); open the PR with labels; post a cost comment (soft cap ~$3); Slack only on failure.
labels: auto-generated, new-api
SCENE 7
Human reviews
Maintainer reads the summary, checks surfaced methods + version bumps, confirms the build, and merges. The wrapper is now in step with core 8.2.0.
done

Where things live

The Hub one, shared

CleverTap/clevertap-wrapper-tooling
  • sync.yml — reusable Conductor ②
  • .github/actions/ — setup, claude-sync, open-pr, build ②
  • tools/diff_native_api.py — ground-truth diff ④
  • prompts/ — the instructions Claude runs ③
  • scripts/ — cost / PR / Slack helpers

The Wrappers many, thin

clevertap-cordova · -flutter · -react-native
  • native-release-sync.yml — the Trigger doorbell ①
  • .claude/skills/ — the wrapper's own know-how
  • bridge code + example apps — what gets edited

The thin-pointer model: the dispatch is a TV remote, not the TV. Fix the logic once in the hub — every wrapper benefits. Each wrapper pins a hub tag (@v1) so a mid-flight hub change can't surprise it.

The vocabulary you'll see in every PR

Triage: what Claude decides per change.

VerdictWhat it meansExample
surface Implement it in the wrapper — write the bridge on all sides.the productive 90% A new public method that maps cleanly across JS / Android / iOS.
skip Deliberately don't expose it.a conscious choice, recorded An internal helper not meant for app developers.
defer Can't safely auto-apply — leave a note for a human.punt, don't break A change needing design judgment the automation shouldn't make.
flag Can't even confirm it — raise a flag instead of guessing.verify or flag, never guess A method named in the changelog it can't find in the source.

The takeaway

The robot does the tedious 90%. The human always reviews.
