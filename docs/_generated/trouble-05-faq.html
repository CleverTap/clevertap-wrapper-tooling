Troubleshooting · 05

FAQ

The most common questions and gotchas, pulled together. If you skimmed everything else, this page is the cheat sheet. Each answer links to the fuller explanation.

1. Does the YAML workflow actually write the wrapper code?

No. The workflow (Layer ②, the Conductor) only sequences and gates steps. The actual code-writing is done by Claude (Layer ③, the Brain), which the conductor hands a written prompt to. This is the single idea beginners get backwards. See the 4-layer onion.

2. I asked to sync but nothing changed. Broken?

Almost certainly not. You probably synced to the version already pinned — Claude correctly no-ops. Pick a target version that differs from the current pin. See failure modes §3.

3. The PR opened with a build-failed label. Is that an emergency?

No — it's by design ("Option A"). Claude's edits didn't compile, so the PR opens anyway. A human pulls the branch, fixes the build, pushes, and merges. See review the PR and failure modes §2.

4. The run died at setup with Invalid keyData. What now?

The GitHub App private key's newlines were mangled when pasted into the secret. Re-paste the exact .pem bytes (e.g. cat key.pem | pbcopy) and verify the key. See prerequisites and failure modes §4.

5. How do I test without spending AI money or touching protected branches?

Use skip_sync=true (builds only, $0 AI), a base_ref baseline branch, or full fork testing per the hub's TESTING.md. See dry-run and skip-sync.

6. The native SDK added a method but the PR didn't surface it. Why?

Could be three things: it was deliberately skipped (internal), it's in flagged_for_review (named in the changelog but unconfirmable), or the regex diff missed it and the recall pass didn't recover it. Check those PR sections first. See when Claude misses an API.

7. Why does the diff tool only catch ~80% of changes?

It uses regex (pattern-matching), not a full parser — a deliberate simplicity trade-off. The remaining ~20% are caught by the changelog recall pass. See when Claude misses an API and Glossary → Regex.

8. Claude "couldn't verify" a method and over-flagged everything. Why?

Most likely it tried to inspect the cached native source with Bash, which is denied outside the working directory. The fix is to use the Read/Grep/Glob tools (not path-restricted) for native-source checks. See source verification failures.

9. The cost number Claude printed doesn't match the cost report. Which is right?

The cost report — it reads the authoritative CLI envelope (total_cost_usd, usage.*). Numbers Claude writes in its own prose can drift and aren't trustworthy. See cost and caps.

10. A run cost more than the $3 cap. Did it stop the PR?

No. The cap is soft — it posts an informational comment but never aborts a run (so it can't leave a half-finished PR). See cost and caps.

11. I updated a skill but the run still behaved the old way. Why?

Skills must be committed to the base_ref branch the run syncs from — Claude discovers .claude/skills/ from the checked-out branch. Commit your skill changes before the run picks them up. See source verification failures.

12. My retry bails immediately saying the branch exists. How do I get unstuck?

A previous failed run left a task/release_<name> branch. Re-run with a unique release_name to avoid the collision. See trigger a sync and failure modes §4.

Check yourself
1. Name the two "this looks broken but it's by design" behaviors.

(1) Syncing to the already-pinned version → Claude no-ops (Q2). (2) A build-failed PR still opening, for a human to fix (Q3). Both are intended, not bugs.

2. What's the one-line fix for the two most common setup/retry failures?

Invalid keyData → re-paste the private key with intact newlines (Q4). Branch-exists on retry → use a unique release_name (Q12).
