Wrapper-Sync — How One Click Becomes a Pull Request

When CleverTap ships a new native SDK, the language wrappers (Cordova, Flutter, React Native…) have to catch up. Wrapper-Sync does that catch-up automatically. A maintainer opens a wrapper repo, picks the versions to sync to, and clicks one button. Behind the glass, a YAML conductor sequences the work, hands a written prompt to an AI that reads exactly what changed and rewrites the bridge code, and a plain Python diff tool keeps the AI honest with cold facts. The output is always the same shape: a pull request a human reviews and merges.

◎ The 4-layer onion

The one model that explains everything. A maintainer's click travels down the four layers; a finished pull request travels back up. The conductor (②) is pure plumbing — order and gates. The intelligence lives in layer ③: the YAML hands a written prompt to an AI, which leans on the non-AI diff tool (④) for facts. Think librarian (④, only reports what's on the shelf) vs. editor (③, decides what to publish).

A single click fans out into a fixed choreography. The conductor mints a bot token and branches, then runs a pre-sync build gate before spending any AI budget. Only then does it hand a prompt to the Brain, which runs the diff tool, triages each change, and edits all three sides of the bridge. The conductor rebuilds, packages a commit, and opens the PR — the human approval step is never removed.

The conductor's cleverness is not domain knowledge — it is gating. The diamonds are the if: conditions that decide which steps run based on what passed or failed.

The pre-sync gate is the money-saver: if the example app is already broken before any change, the run aborts before Claude is invoked, so no AI budget is wasted on a doomed run. 03 Ground truth: the diff tool

The one deterministic, non-AI component. Given two native versions, diff_native_api.py reports exactly what changed — no opinions, no judgment. It runs in four stages.

Four stages: acquire both versions' source, extract the public-method surface, diff old vs new, and render a machine file (diff.json) plus a human file (diff.md). The extractor finds methods with regex, which catches roughly 80% of changes; the remaining ~20% are swept up by Claude's recall pass — re-reading the native changelog. An AI can hallucinate a method; this tool cannot, so Claude is told to trust it as fact and to source-verify any symbol before using it.

The hub (clevertap-wrapper-tooling) holds the reusable conductor, the diff tool, and the claude-sync action; each wrapper repo carries only a thin dispatch file. Two external services do the privileged work: a GitHub App mints a short-lived installation token (so no long-lived credentials are stored), and the Anthropic API serves the model. The PR always lands back in the wrapper repo for a human to review and merge.
＊ Key terms

The vocabulary that ties the diagrams together.

Triage — surface / skip / defer / flag

For every change the diff tool reports, the Brain (③) makes one call. Surface = wrap it now (write the JS/Android/iOS bridge, demo it in the example apps). Skip = intentionally not wrapped (internal, irrelevant, or out of scope). Defer = worth wrapping but punted to a later pass. Flag = genuinely uncertain — leave it for the human reviewer to decide on the PR.

Source verification

The diff tool's regex catches ~80% of changes. Because an AI can hallucinate a method, Claude must read the real native source to confirm any symbol exists before wrapping it. The remaining ~20% are recovered by re-reading the changelog (the recall pass).

Pre-sync gate

A build of the example app before any edits. If it is already broken, the run aborts before the AI is invoked — failing fast and cheap, with no AI cost spent.

Non-fatal post-sync build

A rebuild with the AI's edits. If it fails to compile, the PR still opens — it just gets a build-failed label and a re-asserted failure. A draft a human can fix beats no PR.

Librarian vs. editor

The diff tool (④) is the librarian — it only tells you what is actually on the shelf. Claude (③) is the editor — creative, but anchored to a librarian who never makes things up.

Always a pull request

The output is always the same shape: a PR a human reviews and merges. The automation does the tedious 90%; the human approval step is never removed.
